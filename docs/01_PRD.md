# JoB_Find 产品需求文档

## 1. 背景

大学生在实习和校招投递中通常面临三个重复劳动：

1. 从招聘平台搜索和筛选大量岗位。
2. 根据岗位描述调整简历重点。
3. 跟踪投递状态、简历版本、沟通记录和异常。

JoB_Find 计划提供一个本地部署、开源、可配置的智能体框架，帮助用户在 BOSS 直聘网页端完成岗位筛选、简历定制、审批和投递记录管理。用户自行配置 LLM API 调用地址、密钥和模型，项目只提供框架、默认流程和安全边界。

## 2. 产品目标

### 2.1 核心目标

- 面向实习和校招用户，提供本地化岗位投递辅助框架。
- 支持用户上传基础简历和照片，并通过 Agent 对话补全真实经历库。
- 支持用户选择目标岗位、城市、硬条件、岗位扩展规则和投递策略。
- 使用 AI 对岗位进行筛选、评分和风险识别。
- 使用固定 DOCX 模板生成岗位定制简历，并导出 PDF。
- 简历长度控制在 2 页以内。
- 使用独立审核 Agent 检查简历真实性、风险和越界表达。
- 支持高匹配低风险岗位在用户开启自动投递后自动执行。
- 对中等匹配、低匹配、中等风险及以上岗位要求用户确认。
- 跳过已沟通 HR。
- 记录所有岗位、评分、简历版本、投递动作、审批结果和异常日志。
- 支持投递记录导出，便于复盘。

### 2.2 开源目标

- 采用 MIT 许可证。
- 本地单用户部署为第一优先级。
- 用户可自由替换 OpenAI-compatible 模型服务。
- 优先复用成熟开源项目和库，避免重复造轮子。
- 不复制 GPL 项目的代码进入主仓库，避免许可证传染风险。

## 3. 用户画像

### 3.1 主要用户

- 大学生、研究生、应届生。
- 目标岗位为实习、校招、管培或初级岗位。
- 希望批量筛选岗位，但仍需要控制简历真实性和投递风险。
- 具备基本本地部署能力，能填写 API Key 和模型地址。

### 3.2 次要用户

- 想研究招聘自动化、Agent 编排、简历生成的开发者。
- 想二次开发浏览器投递助手的开源贡献者。

## 4. 第一版范围

### 4.1 In Scope

- BOSS 直聘网页端。
- 本地单用户部署。
- 用户上传基础简历和照片。
- 固定 DOCX 简历模板。
- 简历解析和经历库补全对话。
- 岗位硬条件和软条件配置。
- 岗位扩展搜索配置。
- 岗位 AI 筛选和风险评分。
- AI 定制简历。
- DOCX 和 PDF 生成。
- 独立 Agent 审核。
- 用户审批和自动投递开关。
- 跳过已沟通 HR。
- 异常弹出和日志记录。
- SQLite 本地数据库。
- CSV/Excel 友好的投递记录导出。

### 4.2 Out of Scope

- 多用户 SaaS。
- 云端账号体系。
- 移动端 App 自动化。
- 非 BOSS 直聘平台。
- 验证码绕过。
- 登录绕过。
- 平台风控规避。
- 简历内容造假。
- 无限自动化海投默认开启。

## 5. 核心用户流程

### 5.1 初始化流程

1. 用户本地启动 JoB_Find。
2. 用户填写 LLM 配置：`base_url`、`api_key`、`model`。
3. 用户上传基础简历文件和照片。
4. 系统解析简历，生成初始经历库。
5. ProfileAgent 与用户对话，补全项目、实习、技能、奖项、偏好和真实性边界。
6. 用户选择目标岗位、城市、岗位扩展策略、硬条件和自动投递策略。
7. 系统保存配置和经历库。

### 5.2 岗位筛选流程

1. 用户打开 BOSS 直聘网页并完成登录。
2. 浏览器脚本读取搜索页岗位列表和岗位详情。
3. BrowserExecutor 将岗位结构化信息发送给本地后端。
4. ScreeningAgent 进行硬条件过滤、软条件评分、风险识别。
5. DiscoveryAgent 根据配置提出可扩展岗位方向。
6. 系统将岗位分为：
   - 自动准备；
   - 需要人工确认；
   - 跳过。

### 5.3 简历定制流程

1. ResumeAgent 根据岗位 JD 和真实经历库生成岗位定制简历草稿。
2. DocumentRenderer 使用固定 DOCX 模板渲染简历。
3. 系统检查页数，保证不超过 2 页。
4. 系统生成 PDF。
5. 系统生成修改说明和差异摘要。
6. AuditAgent 独立审核简历是否真实、是否夸大、是否与岗位匹配。
7. ApprovalAgent 决定自动投递、人工确认或跳过。

### 5.4 投递流程

1. 对符合自动投递规则的岗位，在用户开启自动投递时执行投递。
2. 对中等匹配或有风险岗位，展示审批页面。
3. 用户可以同意投递、拒绝投递、要求重新生成简历或调整规则。
4. BrowserExecutor 执行上传简历、发送打招呼语和点击投递。
5. 遇到验证码、登录失效、页面异常或上传失败时，立即暂停该任务并记录错误。
6. ApplicationLogger 记录完整事件链路。

## 6. 功能需求

### FR-1 LLM 配置

- 用户可以通过配置文件填写模型服务。
- 用户可以通过网页控制台填写模型服务。
- 支持 OpenAI-compatible API。
- 配置项包括 `base_url`、`api_key`、`model`、`timeout`、`temperature`。
- 配置保存于本地，不上传云端。

### FR-2 简历上传与解析

- 支持 DOCX 和 PDF 基础简历上传。
- 支持上传照片。
- 能解析教育经历、技能、项目、实习、奖项、自我评价等结构。
- 解析失败时应抛出明确错误，并允许用户手动补充。

### FR-3 真实经历库

- ProfileAgent 与用户对话补全经历库。
- 经历库需要标注证据来源、熟练度、可改写范围和禁止扩写范围。
- 任何简历生成 Agent 都只能使用经历库中的真实内容。

### FR-4 岗位配置

- 用户可以配置岗位关键词、城市、岗位类型、薪资、行业偏好、公司黑名单、关键词黑名单。
- 硬条件可以开启、关闭或由系统建议补充。
- 岗位扩展规则可配置，例如从 AI 产品扩展到数据产品、产品助理、AIGC 运营。

### FR-5 岗位采集

- MVP 通过 Tampermonkey 读取 BOSS 直聘网页 DOM。
- 采集字段包括职位名、公司、城市、薪资、岗位描述、要求、HR 状态、是否已沟通、链接。
- 已沟通 HR 默认跳过。

### FR-6 岗位筛选

- ScreeningAgent 输出匹配分、风险分、推荐等级、匹配理由、风险理由。
- 硬条件失败应直接标记跳过，除非用户关闭对应硬条件。
- 结果必须结构化保存。

### FR-7 简历定制

- ResumeAgent 使用固定 DOCX 模板生成简历内容。
- 输出 DOCX 和 PDF。
- 简历长度必须控制在 2 页以内。
- 生成差异摘要，说明修改了哪些模块和为什么修改。
- 不允许新增不存在的实习、项目、技术栈、奖项、公司、量化成果。

### FR-8 独立审核

- AuditAgent 必须独立于 ResumeAgent 执行。
- 审核内容包括真实性、夸大风险、岗位契合度、页数、格式、敏感表述。
- 审核结果必须影响 ApprovalAgent 的最终决策。

### FR-9 审批与投递

- 默认关闭自动投递。
- 用户可开启高匹配低风险自动投递。
- 中等匹配、低匹配、中等风险及以上必须用户确认。
- 支持用户授权后投递。
- 投递执行过程中异常必须弹出或记录日志。

### FR-10 日志与导出

- 记录岗位详情、评分、风险、生成文件、审批、投递状态、异常。
- 支持导出 CSV。
- 支持导出 Excel 友好格式。

## 7. 非功能需求

### NFR-1 本地优先

- 所有简历、照片、经历库、投递记录默认保存在本地。
- 不需要账号系统。
- 不需要云端数据库。

### NFR-2 可配置

- 模型服务可替换。
- 阈值可配置。
- Agent 可启用、关闭或替换实现。
- 岗位扩展策略可配置。

### NFR-3 可追溯

- 每个岗位必须有可追溯记录。
- 每份简历必须记录来源岗位和生成时间。
- 每次 AI 输出必须可回放或保留摘要。

### NFR-4 错误可诊断

- Agent 执行错误必须抛出结构化错误。
- 浏览器自动化错误必须记录页面、步骤、原因和建议处理方式。
- 用户能从日志判断失败发生在哪个模块。

### NFR-5 合规边界

- 不绕过验证码。
- 不绕过登录。
- 不规避风控。
- 不默认无限自动投递。
- 不生成虚假简历内容。

## 8. 自动投递决策规则

默认建议：

```text
自动投递：
匹配分 >= 85
且 风险分 <= 20
且 所有启用硬条件通过
且 AuditAgent 审核通过
且 未与该 HR 沟通过
且 用户开启自动投递

人工确认：
匹配分 60-84
或 风险分 21-50
或 简历修改幅度较大
或 岗位扩展方向偏离用户主目标
或 公司/岗位信息不足

跳过：
启用硬条件失败
或 风险分 > 50
或 明显不相关
或 明显培训/销售/外包风险
或 要求经验明显超出用户背景
或 已沟通过 HR
```

## 10. 验收标准

### 10.1 MVP 验收

- 用户能在本地启动后端服务。
- 用户能填写或加载 LLM 配置。
- 用户能上传基础简历和照片。
- 系统能生成结构化经历库草稿。
- ProfileAgent 能与用户对话补充经历库。
- 浏览器脚本能读取 BOSS 直聘当前搜索页岗位列表。
- 系统能对至少 10 个岗位输出结构化评分和风险判断。
- 系统能对推荐岗位生成 DOCX 和 PDF 简历。
- 生成简历不超过 2 页。
- AuditAgent 能独立输出真实性审核结果。
- ApprovalAgent 能把岗位分为自动投递、人工确认、跳过。
- 默认自动投递关闭。
- 用户确认后系统能辅助完成一次投递动作。
- 已沟通 HR 能被识别并跳过。
- 遇到验证码或登录异常时系统暂停并记录日志。
- 投递记录能导出为 CSV。

### 10.2 稳定版验收

- 支持网页控制台修改配置。
- 支持岗位扩展搜索策略。
- 支持批量审批面板。
- 支持导出 Excel 友好文件。
- 支持按公司、城市、岗位、状态筛选投递记录。
- 支持失败任务重试。
- 支持简历版本和岗位记录一一对应。
- 支持 Agent 输出回放或调试视图。

### 10.3 质量验收

- 所有 Agent 输出必须符合约定 JSON Schema。
- 所有关键流程必须有单元测试或集成测试。
- 所有自动化执行步骤必须写入日志。
- 不允许在浏览器端保存明文 API Key。
- 不允许默认开启自动投递。
- 不允许任何模块生成未在经历库中出现过的事实性经历。

## 11. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 平台页面结构变化 | DOM 解析失败 | 封装 BrowserExecutor，增加选择器配置和失败日志 |
| 简历生成虚构内容 | 用户信誉风险 | 真实经历库 + AuditAgent + diff + 人工确认 |
| 自动投递误投 | 账号和求职风险 | 默认关闭自动投递，阈值和风险门槛控制 |
| 验证码或登录异常 | 流程中断 | 不绕过，暂停并提示用户处理 |
| LLM 输出不稳定 | 评分和简历质量波动 | JSON Schema 校验、重试、审计 Agent、日志回放 |
| PDF 转换环境差异 | 文件生成失败 | 优先 LibreOffice headless，提供 fallback 和错误提示 |
| 开源许可证冲突 | 影响项目发布 | 只复用 MIT/Apache/BSD 兼容库；GPL 项目只参考思路 |

### 11.1 BOSS 平台层补充约束

BOSS 直聘闭环不是普通 API 调用链。岗位详情、已沟通状态、打招呼、简历投递解锁、简历上传/选择都依赖用户当前浏览器页面状态。后续开发必须把 BOSS 侧实现为 `BossAdapter / BrowserExecutor`，而不是把平台内部接口直接耦合进 Agent 层。

已知问题、状态机和数据对象见 [06_BOSS_PLATFORM_LOGIC.md](06_BOSS_PLATFORM_LOGIC.md)。

## 12. 成功指标

- 单个用户从上传简历到完成首个投递的路径可跑通。
- 岗位筛选能明显减少人工阅读 JD 的数量。
- 生成简历需要人工大改的比例逐步下降。
- 所有投递动作均有可追溯记录。
- 用户能清楚看到系统为何推荐、跳过或要求人工确认某个岗位。
