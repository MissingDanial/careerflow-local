# CareerFlow Local 产品需求文档

> 当前基线：M17.1，SQLite schema v18。历史文档中的自动打招呼、自动上传和自动投递设想不属于当前产品范围。

## 1. 产品定位

CareerFlow Local 是面向实习、校招和早期职业用户的本地求职工作台。它把重复且适合自动化的工作收敛为四个阶段：

~~~text
完整 JD 采集
-> 风险门禁与岗位匹配
-> 定制简历生成与审核
-> 打开 BOSS 岗位页面并记录人工进度
~~~

系统负责读取用户当前已登录、已渲染的 BOSS 页面，完成本地筛选、Agent 编排、DOCX 输出、日志和状态管理。用户负责在 BOSS 页面内打招呼、选择或上传简历以及最终投递。

产品不是自动投递器，不绕过登录、验证码、安全验证或平台风控。

## 2. 核心目标

- 在不打开 DevTools 的情况下采集当前 BOSS 搜索页岗位和完整 JD。
- 用多个“意向岗位队列”隔离产品、算法、Agent 开发等不同求职方向。
- 对全局岗位和 application 去重，同时允许同一岗位加入不同意向队列。
- 先执行用户可配置的风险门禁，再对可接受岗位计算匹配度。
- 基于持久化个人画像和完整 JD 生成不超过两页的定制 DOCX。
- 用 Fit、Claim 和 Audit 门禁约束简历质量，保留失败原因和修订记录。
- 一键打开目标岗位页面，由用户完成外部动作并手动记录进度。
- 所有画像、岗位、简历、模型配置和日志默认保存在本机。

## 3. 产品边界

### 3.1 当前范围

- Chrome Manifest V3 扩展。
- Node.js 24 本地后端和 SQLite v18。
- Native Messaging Host 启动或探测固定的本地后端。
- BOSS 当前登录页面的岗位列表和详情读取。
- 自动打开当前已加载岗位详情，逐项补齐 JD。
- 多意向岗位队列、队列切换、软删除和成员移除。
- 完整 JD 岗位的批量筛选、信任后重筛和批量简历生成。
- ProfileAgent 专属入口、简历文本提取、多轮对话、事实草稿确认和画像版本化。
- OpenAI-compatible 模型配置、脱敏展示和连接测试。
- DOCX 输出目录配置、渲染 QA、Fit、Claim、Audit 和修订闭环。
- 打开岗位详情 URL，以及人工状态 NOT_CONTACTED、GREETED、APPLIED。
- Workflow event、Agent run、异常队列和时间线。

### 3.2 当前明确不做

- 自动点击 BOSS 的打招呼或发送按钮。
- 自动选择或上传本地简历文件。
- 自动点击投递、确认或提交。
- 获取、转发或持久化 BOSS Cookie/session。
- 绕过登录、验证码、安全验证、频率限制或反爬策略。
- 把模型评分、Shadow 评审或本地审批当成外部动作授权。
- 多用户 SaaS、云端账号体系和云端简历数据库。
- 生成不存在的经历、项目、公司、奖项或量化结果。

历史 M14 单岗位真实打招呼 canary 只作为默认关闭、普通界面不可见的兼容实验保留，不属于当前验收范围，也不能扩展为上传或投递能力。

## 4. 用户信息架构

### 4.1 插件弹窗

弹窗只保留高频采集动作：

- 后端运行状态和“启动后端”。
- 当前采集目标队列。
- “开始岗位信息采集”“暂停”“重试”。
- 完整岗位、待补 JD 和失败数量。
- 当前队列“清理待补 JD”。

最大本地缓存岗位数、最多补齐岗位数和自动点击间隔属于低频配置，放在设置页。

### 4.2 工作台

工作台按当前意向队列展示四个阶段：

1. **完整 JD**：只展示可用于筛选的完整岗位；支持多选并移出当前队列。
2. **岗位筛选**：填写风险门禁并批量运行；被过滤岗位可查看原因，也可显式添加队列级信任后重新评估。
3. **定制简历**：对筛选通过岗位批量运行简历闭环；展示成功/失败数量、岗位详情和 DOCX 本地路径；允许设置输出目录。
4. **人工联系与投递**：展示岗位和简历路径；打开 BOSS 岗位页；按未联系、已打招呼、已投递筛选并记录人工状态。

顶部统计始终受当前队列影响。切换队列后，列表、统计、筛选和批量操作都必须切换到同一作用域。

### 4.3 个人经历

个人经历是 ProfileAgent 的专属入口：

- 上传 DOCX、PDF、TXT 或 Markdown 简历并抽取文本。
- 根据项目内 career retrospective Skill 与用户多轮对话。
- 将模型建议保存为待确认事实草稿。
- 用户确认后才写入正式经历、技能和约束。
- 用户点击整理后生成并版本化 career_agent_context.md。
- 提供查看当前画像、对话历史、冲突和待确认草稿的入口。

ProfileAgent 是持久化上游画像沉淀器，不在每个岗位任务中重跑或重置。

### 4.4 设置

普通设置页保留：

- 后端地址、同步开关和采集参数。
- 模型 Base URL、API 类型、模型名、超时、推理强度和 API Key。
- 模型连接测试与脱敏状态。
- 风险门禁、简历输出目录等工作流配置。
- 最近异常和 Agent/workflow 日志。

旧简历实验工具、真实动作授权和 Shadow 评审仅保留在隐藏兼容区，不面向普通用户。

## 5. 核心流程

### 5.1 首次启动

1. 用户安装扩展和 Native Messaging Host。
2. 用户从插件弹窗启动本地后端。
3. 用户在设置页配置模型服务并完成连接测试。
4. 用户上传基础简历，在个人经历入口补全并确认画像。
5. 用户创建或选择一个意向岗位队列。

Native Host 只接受 STATUS 和 START_BACKEND，不能执行任意命令，也不接触 BOSS 页面。

### 5.2 岗位采集

1. 用户在 BOSS 完成登录和搜索条件选择。
2. 用户在插件弹窗选择采集目标队列并开始采集。
3. 内容脚本读取当前已加载岗位卡片，按页面可见顺序打开详情补齐 JD。
4. 完整岗位自动同步到本地后端；缺 JD 和选择器失败保留诊断信息。
5. 同一岗位重复出现时更新全局岗位快照，不新建重复 application。
6. 同一岗位已是当前队列的 active 成员时跳过重复添加。
7. 同一岗位在另一个队列采集时，可复用全局岗位/application，并新增该队列成员关系。

主工作台只展示完整 JD。待补 JD 可继续补齐，也可一次性从当前队列软移除。

### 5.3 岗位筛选

1. 用户配置排斥方向，例如销售、直播、培训或其他自定义风险关键词。
2. JobRiskGate 先执行确定性门禁。
3. 高风险命中直接标记跳过，不再计算常规匹配分。
4. 未命中门禁的岗位进入 ScreeningAgent，输出匹配分、风险分、推荐和理由。
5. 用户可对误筛岗位添加当前队列级信任并重筛。

信任只绕过当前队列的方向门禁，不等于岗位自动通过，也不影响同一岗位在其他队列的规则。

### 5.4 定制简历

对筛选通过岗位运行本地 LangGraph：

~~~text
冻结画像与 JD
-> ScreeningAgent
-> ResumeAgent
-> ResumeFitEvaluator
-> ClaimVerifier
-> 必要时 ResumeRevisionAgent
-> 重新检查 Fit 与 Claim
-> AuditAgent
-> DocumentRenderer 输出 DOCX
~~~

约束：

- 只使用正式确认的画像事实。
- 不覆盖旧简历版本。
- unsupported claim 必须修订或阻断。
- JD 必要项缺失和渲染 QA 失败必须可见。
- 默认控制在两页以内。
- 任一节点失败都写入 Agent run、workflow event 和错误队列。

### 5.5 人工联系与投递

1. 用户在工作台选择已生成简历的岗位。
2. 系统打开规范化后的 BOSS 岗位 URL，并展示 DOCX 本地路径。
3. 用户在 BOSS 页面人工打招呼、沟通、上传简历和投递。
4. 用户回到工作台记录 NOT_CONTACTED、GREETED 或 APPLIED。

manual_status 是用户记录的外部进度，不自动修改 applications.status，也不声称 BOSS 已接受某次投递。

## 6. 数据与去重规则

### 6.1 全局实体

- jobs：规范化岗位主记录。
- job_snapshots：每次采集的字段快照。
- applications：同一岗位的本地工作流记录。
- companies：可增量补齐的公司信息。
- resume_versions：岗位定制简历版本。
- agent_runs、workflow_events：执行与诊断记录。

岗位 URL、岗位 ID 和稳定字段共同参与全局去重。刷新 BOSS 页面或重复采集不能产生重复岗位/application。

### 6.2 队列实体

- application_queues：意向岗位队列。
- application_queue_items：队列与 application 的多对多关系。

规则：

- 新建队列为空。
- 默认队列不可删除。
- 删除自定义队列使用归档，不硬删除岗位、筛选、简历或日志。
- 从队列删除岗位只把成员关系标记为 REMOVED。
- 清理待补 JD 只作用于当前队列。
- 信任标记保存在队列成员上。

### 6.3 状态分离

- applications.status：本地采集、筛选、简历和证据状态，只能由 ApplicationTransitionService 按合法边和证据更新。
- applications.manual_status：用户记录的外部联系/投递进度，只能由窄接口更新。

任何普通 store 更新、错误处理或 UI 操作都不能绕过这两个所有权边界。

## 7. 模型与降级

- hybrid：真实模型加确定性风险、证据和审核门禁；模型失败时停止。
- auto：先调用模型，失败后记录原因并显式降级到规则实现。
- llm：严格模型输出，无规则降级。
- rules：离线恢复和确定性 CI。

模型输出必须通过 Zod Schema 和证据校验后才能写入业务表。API Key 只保存于本地后端配置文件，前端和 API 只返回是否已配置及脱敏信息。

## 8. 错误与可观测性

每个长任务应提供：

- 开始、节点进度、完成和失败 hook。
- application timeline。
- Agent run、模型延迟、重试和降级信息。
- 页面 URL、选择器计数、失败步骤和登录/验证信号。
- 可折叠的最近异常和待补 JD 列表。
- 显式重试或纠正入口。

把错误标记为 RESOLVED 或 IGNORED 只更新本地诊断元数据，不自动重试、不推进状态，也不触发 BOSS 动作。

## 9. 验收标准

### 9.1 M17/M17.1 主流程

- 插件可探测并一键启动固定本地后端。
- 后端模型配置可保存、脱敏读取和连接测试。
- 用户可新建、切换和归档意向岗位队列。
- 新队列为空，默认队列不可删除。
- 重复岗位不会生成重复全局岗位或 application。
- 当前队列可批量移出岗位和一次性清理待补 JD。
- 工作台主列表只显示完整 JD 岗位。
- 风险门禁命中岗位可查看原因，显式信任后可重筛。
- 筛选通过岗位可批量生成 DOCX，并显示成功/失败和本地路径。
- 用户可打开岗位页并记录未联系、已打招呼、已投递。
- 简历上传、ProfileAgent 对话、事实确认和画像版本化可独立完成。
- 异常与 workflow 日志可查看、折叠和纠正。
- 系统不执行真实打招呼、上传或投递。

### 9.2 质量与安全

- 所有数据库迁移可从历史 schema 前滚，并在失败时恢复。
- 生产迁移前自动备份，迁移前后岗位/application 数量不丢失。
- application 状态只能经统一 transition service 更新。
- API、日志和测试输出不包含完整 API Key、Cookie 或 token。
- 桌面和 390px 宽度下无水平溢出或控件重叠。
- npm run check、npm run test:workflow 和 npm run test:ci 全部通过。

## 10. 成功指标

- 完整 JD 采集成功率。
- 同队列重复岗位跳过率和全局去重准确率。
- 风险门禁召回率、岗位排序质量和误筛纠正率。
- DOCX 成功率、两页合规率、Claim 支撑率和 Audit 通过率。
- 从完整 JD 到可人工投递 DOCX 的平均耗时。
- 用户人工状态记录覆盖率。
- 可定位并纠正的失败比例。

真实面试率、回复率和 offer 率需要真实用户长期数据验证，当前离线评测不能替代这些结果指标。
